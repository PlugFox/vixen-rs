# Database Migration Rules

Read this file before creating any database migration.

## Creating a Migration

```bash
# Generate migration file
cd server && sqlx migrate add <feature_name>

# Run migrations
sqlx migrate run

# Revert last migration (manual, ops-only)
sqlx migrate revert
```

File naming: `YYYYMMDDHHMMSS_feature_name.sql` (auto-generated by SQLx CLI). A matching `.down.sql` MUST be authored by hand for every up file.

## Schema Conventions

### Tables

- Table names: `snake_case`, plural where it reads naturally (`chats`, `chat_moderators`, `verified_users`, `moderation_actions`, `report_messages`, `daily_stats`, `chat_info_cache`, `captcha_challenges`, `spam_messages`).
- Primary key:
  - For Telegram-identified rows: composite PK `(chat_id, user_id)` (`verified_users`, `chat_moderators`).
  - For internal entities: `id UUID PRIMARY KEY DEFAULT uuid_generate_v4()` (`captcha_challenges`, `moderation_actions`).
  - For natural keys: that key (`spam_messages.xxh3_hash BIGINT PRIMARY KEY`).
- Foreign keys to `chats(chat_id)`: `ON DELETE CASCADE` (deleting a watched chat purges its history).
- Timestamps: `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.

### Columns — Telegram-specific

- **Telegram IDs are `BIGINT NOT NULL`.** Never `INTEGER` (overflows on supergroup IDs `-100…`), never nullable when they identify the row owner.
- **xxh3-64 spam hashes are `BIGINT`.** Postgres `BIGINT` is signed; `u64::cast_signed()` / `i64::cast_unsigned()` round-trip cleanly.
- **Per-chat config**: prefer a dedicated `chat_config` table for fields that get queried individually; use a JSONB column on `chats` for wide-but-rarely-queried settings.

### Columns — generic

- `VARCHAR(N)` with explicit length for bounded strings (slugs, language codes).
- `TEXT` for unbounded user-supplied content (reasons, sample messages).
- `SMALLINT` for enum-like values with `CHECK` constraints (e.g. `clown_chance BETWEEN 0 AND 100`).
- `JSONB` for flexible metadata (`chats.settings`).
- `TIMESTAMPTZ` for all timestamps (not `TIMESTAMP`).
- `uuid_generate_v4()` for UUID defaults — requires `uuid-ossp` extension.

### Indexes

- Add indexes for columns used in `WHERE`, `ORDER BY`, `JOIN`.
- Composite `(chat_id, created_at DESC)` for time-windowed chat queries (moderation actions list, daily stats).
- Partial indexes where the predicate is narrow: `CREATE INDEX ... ON captcha_challenges (expires_at) WHERE expires_at > NOW()` is overkill — plain btree is fine.
- Name convention: `idx_{table}_{columns}` for non-default indexes; PRIMARY KEY / UNIQUE constraints get auto-named.

### Constraints

- Always add UNIQUE constraints for business uniqueness.
- For `moderation_actions`: `UNIQUE (chat_id, target_user_id, action, message_id)` — this is the idempotency anchor. Without it, retries can double-action.
- `CHECK` for enum-like values: `CHECK (action IN ('ban', 'unban', 'mute', 'unmute', 'delete', 'verify', 'unverify'))`.
- `CHECK` for ranges: `CHECK (clown_chance BETWEEN 0 AND 100)`.

## Triggers

For auto-updated `updated_at`:

```sql
CREATE TRIGGER trg_<table>_updated_at
    BEFORE UPDATE ON <table>
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

The `update_updated_at()` function is defined in the initial migration.

## Migration safety

- `BEGIN; ... COMMIT;` wrapper unless the statement forbids it (`CREATE INDEX CONCURRENTLY` must be alone).
- `IF NOT EXISTS` / `IF EXISTS` for idempotency.
- Never edit an applied migration. Add a new migration that fixes the issue.
- For long ALTERs on big tables: add nullable column → backfill in batches → add NOT NULL with `NOT VALID` then `VALIDATE CONSTRAINT`.
- Adding a `NOT NULL` column to an existing table → must have a default (otherwise the migration fails on non-empty data).

## Checklist

Before submitting a migration:

- [ ] `.up.sql` AND `.down.sql` both exist; down is the inverse.
- [ ] Telegram IDs are `BIGINT NOT NULL`.
- [ ] FKs to `chats(chat_id)` use `ON DELETE CASCADE`.
- [ ] `created_at` / `updated_at` use `TIMESTAMPTZ`.
- [ ] `updated_at` trigger created where the column exists.
- [ ] Indexes for query paths (FK lookups, list pagination).
- [ ] UNIQUE constraints for business uniqueness (incl. `moderation_actions` idempotency key).
- [ ] CHECK constraints for enum-like values and bounded ranges.
- [ ] [`server/docs/database.md`](../database.md) updated.
- [ ] After applying: `cargo sqlx prepare -- --all-targets` and commit `.sqlx/`.
