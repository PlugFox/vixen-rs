# Internationalization

Russian (`ru`) and English (`en`) are first-class. Every user-facing key MUST exist in both locales. The bot's primary user base is Russian-speaking; missing RU translations are a P1 bug.

## File layout

```
website/i18n/messages/
  en/
    common.yaml
    auth.yaml
    chats.yaml
    moderation.yaml
    reports.yaml
    settings.yaml
    errors.yaml
  ru/
    common.yaml
    auth.yaml
    chats.yaml
    moderation.yaml
    reports.yaml
    settings.yaml
    errors.yaml
```

Each locale has the same set of files; each file has the same set of keys. A linter step (planned for `/website-check`) diffs the locale trees and fails if any key is missing.

## Namespaces

Add new keys to the **existing** file for the feature. Create a new namespace only if the feature is genuinely new.

| Namespace | Use for |
|---|---|
| `common` | Generic actions (save, cancel, delete, retry), shared labels (date, count, status). |
| `auth` | Telegram login flow, sign-in errors, "you are not a moderator". |
| `chats` | Chats list, chat detail, chat title formatting. |
| `moderation` | Action ledger, ban / unban / verify dialogs, audit log filters. |
| `reports` | Report viewer, regenerate dialog, AI summary status. |
| `settings` | Per-chat config form labels and hints. |
| `errors` | User-visible error messages keyed by server's error code (`errors.invalid-init-data`, etc.). |

## Key naming

- Dot-separated, kebab-case segments: `chats.list.empty-state`.
- Prefix by feature / page: `chats.detail.title`, `chats.detail.spam-threshold-label`.
- For errors: `errors.<server-code-lowercased>` вЂ” `errors.moderator-required`, `errors.invalid-init-data`. Mirrors `server/docs/rules/error-handling.md`.

## Plurals

Russian uses `one` / `few` / `many` / `other` per CLDR; English uses `one` / `other`:

```yaml
# en
moderation:
  actions:
    count:
      one: "{n} action"
      other: "{n} actions"
# ru
moderation:
  actions:
    count:
      one: "{n} РґРµР№СЃС‚РІРёРµ"
      few: "{n} РґРµР№СЃС‚РІРёСЏ"
      many: "{n} РґРµР№СЃС‚РІРёР№"
      other: "{n} РґРµР№СЃС‚РІРёСЏ"
```

Use `tp(key, count)` вЂ” picks the right branch from CLDR rules for the active locale.

## Usage

```tsx
import { t, tp } from "@/shared/i18n/i18n";
import { T } from "@/shared/i18n/rich";

// Simple
<h1>{t("chats.list.title")}</h1>

// With interpolation
<p>{t("chats.detail.title", { title: chat().title })}</p>

// Plural
<p>{tp("moderation.actions.count", actions().length)}</p>

// Rich text (with inline elements)
<p>
  <T msg="chats.detail.invite-hint" values={{ botName: BOT_NAME }} />
</p>
```

For `aria-label`, `placeholder`, `title`, `alt` attributes вЂ” use `t()` too, not string literals.

## Active locale

Stored in `localStorage` under `vixen_locale`. Defaults to `Telegram.WebApp.initDataUnsafe.user.language_code` (if WebApp), else browser `navigator.language`, else `en`.

```ts
// shared/i18n/i18n.ts
import { createSignal } from "solid-js";

export const [locale, setLocale] = createSignal<"en" | "ru">(detectInitialLocale());

function detectInitialLocale(): "en" | "ru" {
  const stored = localStorage.getItem("vixen_locale");
  if (stored === "en" || stored === "ru") return stored;
  const tg = window.Telegram?.WebApp?.initDataUnsafe?.user?.language_code;
  if (tg?.startsWith("ru")) return "ru";
  if (navigator.language.startsWith("ru")) return "ru";
  return "en";
}

export function setLocaleAndPersist(l: "en" | "ru"): void {
  setLocale(l);
  localStorage.setItem("vixen_locale", l);
  document.documentElement.lang = l;   // accessibility hint for screen readers
}
```

## Code generation

`bun run i18n:gen` walks `i18n/messages/`, validates that both locales have the same keys, and emits a typed surface under `src/shared/i18n/generated/` (gitignored, regenerated on build).

The typed surface gives `t()` autocomplete + compile-time error on missing keys:

```ts
import { common, chats } from "@/shared/i18n/generated";

t(common.save);          // вњ“
t(chats.list.title);     // вњ“
t("chats.list.tytle");   // вњ— TS error: typo
```

## RU vs EN length

RU strings are on average ~30% longer than EN. Don't choose layout sizes that barely fit the English string вЂ” they will break on translation. Test both locales for any new text surface.

Specific places where this matters:

- Tab labels (try to keep в‰¤ 12 chars in RU).
- Button text (avoid wrap at common widths).
- Form labels above narrow inputs.
- Toast messages (short вЂ” avoid horizontal scrollbars).
